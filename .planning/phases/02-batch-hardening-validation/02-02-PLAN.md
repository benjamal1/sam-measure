<objective>
Formalize the two hard-fail data-integrity contracts that already exist functionally in Phase 1's `build_final_csv` (D2-04): CAL-03 (a thread whose session has no ruler calibration hard-fails loudly) and CSV-04 (a thread row that can't be matched to a factor hard-fails the run, never silently skipped/nulled). Extend the discipline to real batch runs: a multi-thread run must refuse to write a partial/corrupted final.csv when ANY thread's session is unmatched, and must not leave a stale prior final.csv implying a good result.

Phase framing: hardening — the raise-before-write mechanism exists (build_final_csv left-merges and raises on NaN px_per_cm before to_csv). This plan gives it requirement-ID-labeled tests, distinct CAL-03 vs CSV-04 messaging, an empty/missing-calibration clear failure, and the no-stale-output guarantee. Directly implements Pitfall 4's guidance ("Hard-fail the join, list the unmatched rows"). No other module changes; file scope is isolated from plans 01 and 03.

Purpose: on a full-batch run a silent join mismatch corrupts the CSV invisibly (Pitfall 4) — this makes every mismatch stop the run loudly.
Output: hardened `build_final_csv.py` + requirement-labeled hard-fail test suite.
</objective>
